# Spec: v0.5.0 — BLE neighbor scanning in the TUI

> **Audience**: a Claude Code cloud agent picking this up cold. Every
> design decision below is **already settled** with the maintainer.
> Implement the spec exactly as written; do not re-litigate scope or
> design unless you find an outright contradiction with the existing
> code, in which case flag it in the PR description and proceed with
> the closest sensible interpretation.

## Goal

Add a second "what's around me" view to the wifiscope TUI: a list of
nearby **BLE (Bluetooth Low Energy) devices** sitting alongside the
existing Wi-Fi scan. The user toggles between the two views with the
`n` key. This answers the question "what electronic devices are
around me right now?" with a population (AirTags, AirPods, smart
watches, smart-home gadgets, BLE keyboards, iBeacons) that the
existing Wi-Fi scan cannot see at all.

Target version: **0.5.0**.

## Settled design decisions

These are **not** open questions. The agent should implement these
choices verbatim.

1. **Layout: A — toggle in place.** The existing "Nearby BSSIDs"
   panel slot is replaced by either a Wi-Fi scan list (existing) or
   a BLE device list (new), depending on the active view. Diagnostics,
   Connection, and Roam log panels are unchanged. Do **not** show
   both lists side-by-side; do **not** create a new modal screen.
2. **Toggle binding: `n`** (mnemonic: "next view"). Add to footer
   bindings. Translated label `n View` / `n 视图`.
3. **Permission model: extend `wifiscope-helper.app`.** Do not use
   the Python `bleak` library. The helper Swift sidecar at
   `helper/Sources/wifiscope-helper/main.swift` already owns Location
   Services TCC; add a second TCC entitlement (Bluetooth) to the
   same bundle. Add a new `ble-scan` subcommand to the helper that
   uses `CoreBluetooth.CBCentralManager` to scan continuously and
   streams JSON Lines (one advertisement event per line) to stdout.
   The Python side spawns this subprocess and reads the pipe in a
   background asyncio task. Rationale: keeps the existing
   "permission isolation" architecture; one bundle, two TCC entries,
   one trust grant flow.
4. **Vendor identification: bundled Bluetooth SIG company-IDs JSON
   plus `make update-vendors`.** Do not hit the network at runtime.
   See "Vendor data" section below for the snapshot procedure.
5. **UUID-rotation fuzzy merge: lightweight, transparent.** Modern
   devices rotate the BLE identifier for privacy, so the same
   physical device appears under multiple UUIDs over time. Merge
   only when **(vendor_id, name, recent_rssi_window) all match
   within tolerance**, never silently fall back. Show a `(merged
   N)` badge when entries are combined. Do **not** try to be clever
   about Apple Continuity payload reverse-engineering; a single
   "Apple (Continuity)" lump is fine.

## Out of scope (do not implement)

- Linux / Windows BLE backend (macOS only this release).
- Bluetooth Classic / BR-EDR scanning (BLE only).
- BLE device connection, GATT read/write, pairing flows.
- Advertising your own device.
- Per-device deep dive / details modal. (Future release.)
- Any external network calls at runtime. The vendor JSON is the
  only "external knowledge" and it ships bundled.

## Architecture

### New module: `src/wifiscope/ble.py`

```python
@dataclass(frozen=True, slots=True)
class BLEDevice:
    identifier: str         # CoreBluetooth UUID, lower-case
    name: str | None        # advertisement local name
    vendor: str | None      # resolved from manufacturer_data prefix
    vendor_id: int | None   # raw Bluetooth SIG company ID
    services: tuple[str, ...]  # advertised service UUID strings
    rssi_dbm: int | None
    is_connectable: bool
    first_seen: datetime
    last_seen: datetime
    ad_count: int           # how many advertisements seen since first_seen


class BLEPoller:
    """Spawns wifiscope-helper ble-scan, parses JSONL stdout, maintains
    a rolling map of {identifier: BLEDevice}, expires entries unseen
    for >self._ttl seconds, exposes an async iterator of snapshots."""

    def __init__(self, helper_path: str, *, ttl_s: float = 30.0,
                 snapshot_interval_s: float = 1.0): ...

    async def events(self) -> AsyncIterator[BLEScanUpdate]: ...

    def stop(self) -> None: ...
```

### Helper Swift addition: `helper/Sources/wifiscope-helper/main.swift`

Add a third subcommand alongside the existing GUI mode and `scan`
subcommand:

```
wifiscope-helper ble-scan
```

When invoked, the helper:

1. Initializes `CBCentralManager`, requests `state == .poweredOn`,
   handles `.unauthorized` by emitting a single JSON error line and
   exiting code 3.
2. On every `centralManager(_:didDiscover:advertisementData:rssi:)`
   callback, prints **one JSON object per line** to stdout, then
   flushes. Schema:

```json
{
  "ts": "2026-05-06T12:34:56.789Z",
  "id": "550E8400-E29B-41D4-A716-446655440000",
  "name": "AirPods Pro",
  "rssi_dbm": -52,
  "is_connectable": true,
  "service_uuids": ["180D", "1812"],
  "manufacturer_id": 76,
  "manufacturer_hex": "4c001907..."
}
```

3. Runs until SIGTERM / parent pipe close. No timeout.

The bundle's `Info.plist` gains
`NSBluetoothAlwaysUsageDescription = "wifiscope scans nearby BLE
advertisements so the TUI can show you what devices are around"`.
The GUI mode (no args) requests both Location Services AND
Bluetooth permissions on launch, so a single Allow click covers both.

### TUI changes: `src/wifiscope/tui.py`

- New `BLEPanel(Static)` widget. Renders a header line plus per-device
  rows. Sort by RSSI desc by default. Show `(merged N)` badge for
  devices folded by the fuzzy merger.
- Columns: ★ (selected; reserved for future) / RSSI (right-aligned)
  / signal-bar / vendor / name / service-summary / last-seen-ago / id-short.
- New `WifiScopeApp._view_mode: str` field, values `"wifi"` or
  `"ble"`. Defaults `"wifi"`.
- New action `action_toggle_view` bound to `n`. Swaps the visible
  panel by mounting/unmounting widgets in the third slot. Subtitle
  updates: `view: wifi` / `视图：Wi-Fi` and `view: ble` / `视图：BLE`.
- BLE poller spawned once on `on_mount`, runs in parallel with the
  Wi-Fi poller. Both update internal state regardless of which view
  is active so toggling is instant.

### Vendor data: bundled JSON + Make target

- `src/wifiscope/data/bluetooth_vendors.json` is committed to the
  repo. Generated by parsing the Bluetooth SIG YAML at
  `https://bitbucket.org/bluetooth-SIG/public/src/main/assigned_numbers/company_identifiers/company_identifiers.yaml`.
  Format: `{"<decimal_company_id>": "<vendor_name>", ...}`. Header
  pseudo-entry `"_meta": {"source_commit": "abc123", "fetched_at":
  "2026-05-06"}` for traceability.
- New `make update-vendors` Make target downloads the YAML, parses
  via Python (`pyyaml` is already a project dep), writes the JSON.
  Idempotent.
- `BLEPoller` loads the JSON once on startup; lookup is dict[int]→str
  with `None` fallback.

### Tests (new file `tests/test_ble.py`)

Mock the helper subprocess at the boundary (same pattern as
`test_helper.py` mocks the existing `wifiscope-helper scan`). Cover:

1. **JSON line parsing happy path** — well-formed JSONL stream
   produces correctly-populated `BLEDevice` instances.
2. **Vendor lookup** — known company ID `76` → `"Apple, Inc."`,
   unknown → `None`.
3. **Service category inference** — UUID `180D` → `"Heart Rate"`,
   `1812` → `"HID"`, unknown UUID passed through as raw.
4. **Decay / TTL** — a device unseen for `>ttl_s` is dropped from
   the snapshot.
5. **Fuzzy merge** — two records with same `(vendor_id, name)` and
   RSSI within ±10 dB merge into one with `ad_count` summed.
6. **Permission denied** — helper exits with code 3 and JSON error;
   `BLEPoller` surfaces the state cleanly without crashing.
7. **Subprocess crash** — helper exits 137 mid-stream; poller stops,
   future iterations yield empty snapshots, no exception bubbles up.
8. **Malformed JSON line** — one bad line in the stream is skipped,
   subsequent lines parse normally.

Add a section to `tests/TESTING.md` (and the Chinese mirror at
`docs/zh/TESTING.md`) describing the new module's coverage targets
and one row per test, in the same format as existing modules.

## Known limitations to accept (do not solve)

- **Apple Continuity / Handoff** advertisements use Apple's company
  ID with an opaque payload. Show as `Apple (Continuity)` or
  `Apple (Generic)`; do not attempt model-level identification.
- **UUID rotation** means the same physical device may appear under
  multiple identifiers over time. The fuzzy merger reduces but does
  not eliminate this. Document the limit in the README.
- **BLE range is short** (~10 m vs Wi-Fi's ~30 m). The list will
  feel "smaller" than the Wi-Fi scan. That is expected; mention in
  the README.
- **macOS hides the underlying MAC**. CoreBluetooth gives only a
  per-host UUID. We cannot do OUI-based identification — vendor
  reverse-lookup goes through the manufacturer-data company ID
  field exclusively.
- **First-launch Bluetooth permission** requires the user to click
  Allow. The existing helper GUI handles this. No new UX flow needed.
