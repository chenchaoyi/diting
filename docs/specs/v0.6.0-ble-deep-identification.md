# Spec: v0.6.0 — BLE deep identification + connected peripherals

> **Audience**: a Claude Code cloud agent picking this up cold.
> Every design decision below is **already settled** with the
> maintainer — implement as written; do not re-litigate scope. If you
> find an outright contradiction with the existing 0.5.0 code base,
> flag it in the PR description and proceed with the closest sensible
> interpretation.

## Goal

Make the BLE view in wifiscope answer two questions it currently
cannot:

1. **What kind of device is this?** Today the user sees rows like
   `Apple, Inc.  (anonymous)  Find My`. Mostly true, mostly useless.
   v0.6.0 parses the public, well-documented BLE advertising formats
   so common devices get an actionable label: `iBeacon`, `AirTag`,
   `Eddystone-URL`, `iPhone`, `Apple Watch`, `HomePod`, `Tile`,
   `SmartTag`, `Swift Pair`.
2. **What's actually connected to my Mac right now?** Today the
   panel only lists advertising peripherals; the AirPods you're
   listening to and the Magic Keyboard you're typing on are *not*
   advertising and so are absent from the list — counter-intuitive
   for a "show me Bluetooth around me" tool. v0.6.0 lists currently-
   connected peripherals in a dedicated section above the advertising
   list.

Target version: **0.6.0**.

## Settled design decisions

These are **not** open questions. Implement verbatim.

1. **Detection scope is limited to public, documented formats.** The
   six categories below carry well-known byte signatures that have
   been reverse-engineered or published by their owners:
     - **iBeacon** — Apple manufacturer-data starts with `0x02 0x15`
     - **AirTag / Find My target** — Apple manufacturer type `0x12`
       OR Find My service UUID `FD5A`
     - **Eddystone** — service UUID `FEAA`, frame type byte 0/1/2/4
     - **Tile** — service UUID `FEED` or `FEEC`
     - **Samsung SmartTag** — service UUID `FD5A` AND vendor=Samsung
       (disambiguates from Apple's Find My on the same UUID)
     - **Microsoft Swift Pair** — Microsoft company ID (0x0006) +
       packet leading byte `0x03`
2. **Apple Nearby Info parsing covers device_class only.** Apple
   manufacturer type `0x10` carries an unencrypted device-class
   nibble (well-documented; see `furiousMAC/continuity` repo). Map
   it to `iPhone` / `iPad` / `Mac` / `Apple TV` / `HomePod` /
   `Apple Watch`. Do **not** attempt to decrypt the rest of the
   payload (lock state, Music-playing, etc.) — they are encrypted
   and any "decoding" is unreliable noise. Do **not** attempt
   per-model identification (iPhone 14 vs 15) — model is not in any
   public ad packet.
3. **Connected peripherals via `retrieveConnectedPeripherals`.**
   The Swift helper enumerates a fixed list of common service UUIDs
   (Audio profiles, HID, Heart Rate, Battery, Find My, Apple
   Continuity) and unions the results. It does **not** issue a
   GATT connection or `readRSSI()` — both would be invasive. The
   list comes back without RSSI; this is fine, the panel renders a
   `—` for the signal column.
4. **Layout B — separate sections.** The BLE panel renders two
   distinct blocks:
     ```
     ── Connected (3) ──
     [—]  Magic Keyboard           Apple, Inc.   HID Keyboard
     [—]  AirPods Pro              Apple, Inc.   Audio
     [—]  Apple Watch              Apple, Inc.   HID
     ── Advertising (24) ──
     -42  AirTag                   Apple, Inc.   Find My  AirTag
     -55  Mi Band 7                Xiaomi        Heart Rate
     ...
     ```
     Connected entries sort alphabetically by name; advertising
     entries continue to sort by RSSI desc. The fuzzy merger applies
     to the advertising block only.
5. **Schema bumped to 3 (forward-compatible).** Helper JSON Lines
   output gains optional fields `type` (iBeacon / AirTag / etc.)
   and `device_class` (iPhone / Mac / etc.); a separate one-shot
   `connected` field surfaces from the connected-peripherals path.
   Python tolerates schema 2 (no deep-ID, no connected list) and
   schema 3.

## Out of scope (do not implement)

- Per-model identification (iPhone 14 vs iPhone 15) — not in public
  advertising data. Anyone claiming to do this is reading
  proprietary GATT services after connecting, which we will not do.
- Decryption of encrypted Continuity payloads (Handoff, AirDrop
  state, lock state, Music-playing). We mention the *fact* of these
  via device_class only.
- Listing peripherals connected to other Macs / phones in range —
  no API.
- GATT connection / pairing / device manipulation. Read-only listing.
- `readRSSI()` against connected peripherals — too invasive.
- Sub-types beyond what Tier 1 covers. If a Bluetooth Mesh / Matter
  / Health Thermometer signature is found, leave it as `(generic)`.

## Architecture

### Helper Swift additions: `helper/Sources/wifiscope-helper/main.swift`

#### Identification

Add a `BLEAdParser` namespace with a single entry point:

```swift
struct BLEDetection {
    let type: String?         // "iBeacon" | "AirTag" | "Eddystone-URL" | ...
    let deviceClass: String?  // "iPhone" | "Mac" | "Apple Watch" | ...
}

func detect(advertisementData: [String: Any]) -> BLEDetection
```

Implementation walks the advertisementData in order:
1. Check service UUIDs for `FEAA` (Eddystone), `FEED`/`FEEC` (Tile),
   `FD5A` (Find My / SmartTag — disambiguate by vendor below).
2. Inspect manufacturer data:
   - Apple (company 76): branch on type byte at offset 2 — `0x02`
     iBeacon, `0x10` Nearby Info → parse device_class, `0x12`
     Find My target → label "AirTag" if owner-paired sub-type
     present, else "Find My target".
   - Microsoft (company 6): branch on first byte — `0x03` Swift
     Pair.
   - Samsung (company 117): if FD5A also present → "SmartTag".
3. Anything else: both fields stay nil.

The resulting fields are added to the per-advertisement JSON line
emitted by `runScanAndDumpJSON`-style emission, alongside the
existing `id` / `name` / `rssi_dbm` / etc.

#### Connected peripherals

Add a once-per-snapshot connected enumeration in the `ble-scan`
loop. Every snapshot interval (or every N advertisement events,
whichever the cleanest pattern), the helper:

1. Calls `retrieveConnectedPeripherals(withServices:)` with a
   pre-defined service-UUID array (see below).
2. For each returned `CBPeripheral`, emits one JSON Line of shape:
   ```json
   {"connected": true, "id": "uuid", "name": "Magic Keyboard",
    "service_uuids": ["1812", "180F"]}
   ```
   Vendor / device_class are NOT inferred for connected peripherals
   (the platform gives us much less to work with than from a fresh
   advertisement); leave them out.
3. The Python side distinguishes `connected:true` lines from
   advertisement lines and routes them into a separate dictionary.

Service UUIDs to enumerate (a deliberately broad union covering the
common bands):
```
0x1108, 0x110A, 0x110B, 0x110C, 0x110D, 0x110E, 0x110F, 0x111E  // Audio
0x1124, 0x1812                                                    // HID
0x180D, 0x180F, 0x1809                                            // Heart Rate, Battery, Health Thermometer
0xFD5A, 0xFE9F                                                    // Find My
0xFEAA, 0xFEED, 0xFEEC                                            // Eddystone, Tile
```

### Python module additions

#### `src/wifiscope/ble.py`

Extend `BLEDevice` with three optional fields (default-aware to
preserve schema-2 compatibility):

```python
@dataclass(frozen=True, slots=True)
class BLEDevice:
    # ... existing fields ...
    type: str | None = None
    device_class: str | None = None
    is_connected: bool = False
```

`update_from_line` learns to:
- Read `type` and `device_class` if present.
- Detect `connected: true` lines and route those into a parallel
  dict keyed on identifier — these never enter `merge_for_display`
  (no RSSI, no UUID rotation; treat them as stable identities).

`BLEScanUpdate` carries both lists:
```python
@dataclass(frozen=True, slots=True)
class BLEScanUpdate:
    devices: list[BLEDevice]            # advertising
    connected: list[BLEDevice]          # currently-connected
    permission_state: str
```

`BLEPoller` reads both streams of JSON Lines and emits the combined
update. The per-snapshot-tick TTL applies to advertising devices
only; connected devices stay in the dict until the next connected-
list snapshot does not include them.

#### `src/wifiscope/tui.py`

`BLEPanel`'s `update_devices` signature gains a `connected` list:
```python
def update_devices(
    self,
    devices: list[BLEDevice],
    connected: list[BLEDevice],
    permission_state: str,
) -> None:
```

Render layout:
```
── Connected (3) ────────────────────────────────────────────────
 —    Magic Keyboard      Apple, Inc.   HID Keyboard
 —    AirPods Pro         Apple, Inc.   Audio
 —    Apple Watch         Apple, Inc.   HID
── Advertising (24) ─────────────────────────────────────────────
-42   AirTag              Apple, Inc.   AirTag · Find My
-55   Mi Band 7           Xiaomi        Heart Rate
...
```

Section headers are dim grey; the connected block is omitted
entirely when empty. A new column shows `type` (and falls back to
service-category if type is None) so iBeacon / AirTag / Eddystone /
device_class show up inline.

The diagnostic panel's existing four lines stay; a fifth line
appears only when connected peripherals exist:
```
Connected     3 peripherals  ·  2 audio  ·  1 HID
```
And the "Categories" line gains the new type breakdown when
non-trivial:
```
Categories    Audio 3 · HID 2 · Heart Rate 1 · iBeacon 4 · AirTag 1
```

## Known limitations to accept (do not solve)

- **Apple Nearby Info encrypted bits are not decoded.** The lock
  state, Music-playing flag, and other interesting bits sit behind
  a per-device key Apple does not publish. Pretending to decode
  them would produce nonsense; we don't.
- **Connected peripherals have no RSSI.** Reading RSSI mid-session
  requires `readRSSI()` against an active connection, which is an
  active perturbation. Sort by name and accept the missing column.
- **Service-UUID enumeration in `retrieveConnectedPeripherals` is
  required.** The hard-coded service list will miss obscure
  peripherals (Bluetooth Mesh nodes, exotic Health Devices). That
  is acceptable for v0.6.0.
- **Apple Continuity carries 30+ subtypes**, most encrypted. We
  surface device_class via type `0x10` only; types `0x05`
  (AirDrop), `0x07` (AirPods), `0x0a` (Magic Switch) etc. are
  detectable but their payloads are uninformative. Skip for now.
- **MAC randomisation persists.** Even with deeper labels, an
  iPhone seen across a 30-minute window may rotate its identifier
  several times and appear as several `iPhone` rows. The fuzzy
  merger now has more signals to work with (`type`, `device_class`)
  but still cannot guarantee a 1:1 mapping.
