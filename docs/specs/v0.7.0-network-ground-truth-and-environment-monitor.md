# Spec: v0.7.0 — network ground truth + environment monitor

> **Audience**: a Claude Code cloud agent picking this up cold. Every
> design decision below is **already settled** with the maintainer.
> Implement the spec exactly as written; do not re-litigate scope or
> design unless you find an outright contradiction with the existing
> 0.6.0 code base, in which case flag it in the PR description and
> proceed with the closest sensible interpretation.

## Goal

Push the existing Wi-Fi data the TUI already collects from "look at
once" into "interpret over time". Three themes, one release:

1. **Network ground truth.** RSSI says nothing about whether your
   link is *actually working*. Add a continuous latency / loss probe
   so a -55 dBm reading still reads as bad when Zoom is glitching.
2. **Beacon IE depth.** CoreWLAN's scan output already contains BSS
   Load, 802.11k/v/r capability bits, and channel utilisation
   measurements that today's helper just discards. Parse them so
   diagnostics can say "your AP is at 78 % utilisation" or
   "3 of 5 same-SSID candidates do not advertise 802.11r" instead
   of guessing from BSSID density.
3. **Environment monitor.** Use rolling RSSI variance (and the
   diagnostic data above) to fire **RF-disturbance events** —
   NEVER claimed as people-counting or motion-detection-with-pose,
   only as "something around me changed". Surface as a 3-line
   in-TUI Events strip, a modal-`m` detail browser, and a headless
   `wifiscope monitor` subcommand for unattended long-runs.

Target version: **0.7.0**.

## Settled design decisions

These are **not** open questions. Implement verbatim.

1. **Latency-probe targets are the gateway plus one auto-detected
   WAN anchor.** ICMP ping at 1 Hz against (a) the user's
   `router_ip` from the Connection panel, (b) the **system's
   currently-configured DNS server**, auto-detected via macOS's
   `SystemConfiguration.framework`. Hardcoding a public anchor
   like `1.1.1.1` is wrong on networks that block it (mainland
   China, some corporate filters) and wastes a probe slot when
   the user's own DNS — which they actually use for every name
   resolution — is the operationally-relevant target.

   **Detection mechanism**: `SCDynamicStoreCopyValue` against the
   key `State:/Network/Global/DNS`. The returned CFDictionary's
   `ServerAddresses` array is the live resolver list — the same
   data `scutil --dns` parses out of the same dynamic store.
   Direct in-process API access (we already depend on
   `pyobjc-framework-systemconfiguration` for the
   `_dynamic_store.py` Wi-Fi fallback path); no subprocess fork,
   no text-parsing, no version-drift exposure. A small
   `_resolve_dns_anchor()` helper in `latency.py` wraps the
   call.

   Resolution order for the WAN anchor:
     1. `WIFISCOPE_LATENCY_WAN_TARGET` env var (explicit override).
     2. First entry of the SCDynamicStore `ServerAddresses` list
        whose IP is **not** equal to the detected gateway IP
        (private vs public makes no difference — many corporate
        networks route DNS through an internal IP, which is still
        a more meaningful test than a public anchor that may be
        blocked).
     3. If the only configured DNS is the gateway itself (the
        common home-router case), **skip the WAN probe** and run
        only the gateway probe. The Diagnostics line then reads
        `Link  gw 12 ms · 0% loss · WAN n/a (DNS == gateway)` so
        the user knows why the second column is missing.
     4. If SCDynamicStore returns no DNS state (no current
        network config, transient boot state), also skip the WAN
        probe.

   The DNS detection re-runs every 60 s so a network switch
   updates the anchor without restarting the TUI. (A future
   refinement could subscribe to SCDynamicStore notifications
   via `SCDynamicStoreSetNotificationKeys` for instant
   change-detection — out of scope for v0.7.0.) We do not probe
   Zoom servers, multiple DNS resolvers, or HTTP endpoints —
   that is feature creep into "internet quality monitor"
   territory.
2. **No raw-socket or sudo-required ICMP.** Use
   `/sbin/ping -c 1 -W 1 -t 1 <ip>` as a subprocess every second.
   macOS allows that without elevation. Parse the standard `time=...`
   output. Failure to parse / non-zero exit = "lost", *not* an
   exception.
3. **BSS Load and 802.11k/v/r come from the helper's existing scan
   subcommand, NOT a new one.** Extend the JSON schema (bump to v3
   for the helper, parallel to the BLE schema-3 bump). Each network
   row optionally adds `bss_load_pct`, `bss_station_count`,
   `supports_802_11r`, `supports_802_11k`, `supports_802_11v`. New
   fields default-absent; the Python ScanResult dataclass adds
   them as `int | None` / `bool | None` so v2 helpers continue to
   work.
4. **Environment stability is a derived statistic, NEVER a count.**
   The Diagnostics panel gains one new line, format:

   ```
   Environment   stable σ 1.2 dB / 60 s
   Environment   ⚠ active σ 7.8 dB / 60 s · last event 12s ago
   ```

   The label is one of three words: `stable` / `active` / `quiet`
   (the last only when occupancy calibration is on). NEVER any
   number-of-people or "motion direction" or "human" wording.
5. **Multi-AP fusion is two distinct modes, picked automatically
   per AP based on signal strength.**
     - Strong same-floor APs (RSSI >= -65 dBm) → *redundancy fusion*:
       a σ spike must occur on N >= 2 APs simultaneously to count
       as a high-confidence event. This filters chip noise.
     - Weak / cross-floor APs (RSSI between -85 and -65 dBm) →
       *spatial channel*: each AP's σ is its own independent event
       lane, labelled with the AP's inventory name from
       `aps.yaml`. So a σ spike on `2F-书房` reads as "motion in
       2F-书房" without any inference from other APs.
     - APs below -85 dBm are too noisy to trust; they are excluded.
6. **Layered monitoring surfaces, three of them.**
     - **Layer 1**: a 3-line `Events` panel in the main TUI,
       always visible, replacing the existing `Roam log` panel
       *with one that carries both kinds of event* (roam + RF
       stir + latency anomaly). Time-ordered, newest at top, ring
       buffer of last 100 events.
     - **Layer 2**: a modal `m` (full-screen) opens an
       `EventsScreen` with the full ring buffer + per-AP σ
       baseline + per-day hourly histogram + a sparkline of σ
       over the last hour.
     - **Layer 3**: a new `wifiscope monitor` subcommand that runs
       headless (no TUI), prints one JSONL event per line to
       stdout, and supports `--out path.jsonl` for file output and
       `--notify` for macOS notification-centre alerts. This is
       the long-run / Home-Assistant integration target.
7. **Single events.jsonl schema for all three layers.** Each event
   is one line:

   ```json
   {"ts": "2026-05-08T14:32:18Z", "type": "rf_stir",
    "magnitude_db": 8.3, "location": "1F-客厅",
    "duration_s": 12, "confidence": "high"}
   {"ts": "2026-05-08T14:34:02Z", "type": "latency_spike",
    "target": "router", "rtt_ms": 412, "loss_pct": 25.0}
   {"ts": "2026-05-08T14:35:11Z", "type": "roam",
    "previous_bssid": "...", "new_bssid": "...",
    "kind": "inter_ap"}
   ```

   Event types: `rf_stir`, `latency_spike`, `loss_burst`, `roam`,
   `link_state` (associated / disassociated). Layer 1 / 2 read
   from the same in-memory ring buffer; Layer 3 streams to its
   sink.
8. **No ML, no trained models, no per-environment calibration
   required for the default flow.** Adaptive baseline = trailing
   5-minute median σ; events fire when current 5 s σ exceeds
   2.5× baseline AND the absolute σ exceeds 3 dB. These are
   thresholds the maintainer picked from the literature and from
   their own RSSI traces; do not "tune" them on synthetic data.
9. **`wifiscope calibrate` is opt-in and stores nothing
   user-identifying.** Records 5 min of RSSI baselines while the
   user explicitly says "the room is empty". Output is a plain
   JSON file at `./wifiscope-baseline.json` (gitignored, never
   bundled with the repo). When present, the Environment line
   gets the `quiet` / `active` qualifier; when absent, it
   continues to use the adaptive baseline approach (so the
   feature works without calibration, calibration just makes it
   better).
10. **Latency probe runs always-on (no opt-in flag).** The
    `_run_tui` startup logic spawns a `LatencyPoller` as a peer to
    `WiFiPoller` and `BLEPoller`. The Diagnostics panel gets ONE
    new line:

    ```
    Link        gw 12 ms · 0% loss · WAN 18 ms · 0% loss · jitter 3 ms
    Link        ⚠ gw 412 ms · 25% loss · WAN unreachable
    ```

## Out of scope (do not implement)

- People counting, occupancy estimates beyond the binary
  `quiet`/`active`. This is the ONE design rule we will not bend.
- Pose estimation, gesture recognition, vital signs, anything that
  belongs in the `docs/explainers/wifi-sensing.md` "out of scope"
  section.
- Internet-quality probing beyond the single public anchor (no
  Speedtest, no Zoom, no DNS resolution timing).
- TCP / HTTP-based latency (only ICMP).
- Linux backend / cross-platform — wifiscope is still macOS only;
  the `LatencyPoller` uses macOS `/sbin/ping` semantics.
- Trend-graph rendering inside the TUI (deferred to v0.8.0; we add
  the events.jsonl schema *as foundation*, but no graph).
- ML models of any kind. Adaptive baseline + threshold only.

## Architecture

### New module: `src/wifiscope/latency.py`

```python
@dataclass(frozen=True, slots=True)
class LatencySample:
    ts: datetime
    target: str         # 'router' | 'wan'
    target_ip: str
    rtt_ms: float | None  # None on loss
    lost: bool


class LatencyPoller:
    """Spawns one `/sbin/ping -c 1 -W 1 -t 1 <ip>` per target per
    second; emits a LatencySample stream. Maintains rolling 60 s
    history per target. The TUI reads aggregates (median rtt, loss
    %, jitter = MAD of rtt) for the diagnostic line; the events
    pipeline reads raw samples for spike detection.
    """

    def __init__(
        self, *,
        gateway_ip: str | None,
        wan_ip: str | None = None,   # auto-detected from scutil --dns
                                      # if None; pass an explicit IP to
                                      # override (env var route).
        interval_s: float = 1.0,
        dns_refresh_s: float = 60.0,
    ): ...

    async def events(self) -> AsyncIterator[LatencySample]: ...
    def aggregate(self, target: str, window_s: float = 60.0) -> dict: ...
    def stop(self) -> None: ...
```

Spike detection thresholds (used by the events pipeline, not by the
poller itself):
- `latency_spike`: rtt > 200 ms AND > 5× the rolling 60 s median
- `loss_burst`: >= 3 of last 5 samples lost

### Helper Swift extensions: `helper/Sources/wifiscope-helper/main.swift`

`runScanAndDumpJSON` is extended to read CoreWLAN's
`informationElementData` for each `CWNetwork` and decode:

- BSS Load IE (Element ID 11): emit `bss_load_pct` and
  `bss_station_count`.
- RM Enabled Capabilities IE (Element ID 70): emit
  `supports_802_11k = true` when the bit is set.
- Mobility Domain IE (Element ID 54): emit `supports_802_11r = true`.
- Extended Capabilities IE (Element ID 127), bit 19: emit
  `supports_802_11v = true`.

Schema bump: top-level `"schema": 3` for the scan output (v2
`schema=2` is still valid; the parser tolerates both). The new
fields are emitted only when the IE is present, so v2 helpers
running v0.7.0 Python (impossible in practice but defensive)
remain forward-compatible.

### TUI changes: `src/wifiscope/tui.py`

Three changes:

1. **Diagnostics panel gains two lines.**
     - `Link` (latency / loss / jitter from LatencyPoller).
     - `Environment` (RSSI σ from the new EnvironmentMonitor, see
       below).

2. **Roam log panel becomes the Events panel.** Same widget slot,
   same height, but accepts `RoamEvent` *and* `RFStirEvent` /
   `LatencySpikeEvent` / `LossBurstEvent` from a unified ring
   buffer. New entries push to the top with their type-tagged
   prefix (`[ROAM]` / `[STIR]` / `[LATENCY]` / `[LOSS]`).

3. **New modal `EventsScreen` on the `m` binding.** Full-screen
   browser of the same ring buffer, sortable by time, filterable
   by type via subkeys (1/2/3/4). The bottom of the modal shows
   per-AP σ baseline statistics so the user can see why events
   fire. `m` opens; `m` or Esc closes.

### Environment monitor: `src/wifiscope/environment.py`

```python
class EnvironmentMonitor:
    """Maintains rolling RSSI history per BSSID, computes σ over a
    sliding window, fires RFStirEvent when the threshold is breached.

    Per-AP modes (auto-classified by RSSI):
      - rssi >= -65 dBm  → 'co_located' (redundancy fusion)
      - rssi -65..-85    → 'spatial_channel' (independent lane)
      - rssi < -85       → 'ignored' (too noisy)

    The MA-AP roster is extracted from inventory.yaml + auto-cluster
    labels so events carry a meaningful `location` string.
    """

    def __init__(
        self, *,
        inventory: NetworkInventory,
        baseline_window_s: float = 300.0,
        spike_window_s: float = 5.0,
        spike_ratio: float = 2.5,
        spike_min_db: float = 3.0,
    ): ...

    def ingest(self, bssid: str, rssi_dbm: int, now: datetime) -> None: ...
    def fire_events(self, now: datetime) -> list[RFStirEvent]: ...
    def baseline_summary(self) -> dict[str, dict]: ...  # for modal
```

The poller calls `ingest()` for every Wi-Fi connection update AND
every neighbour BSSID seen in scan results. `fire_events()` returns
zero-or-more events to push into the ring buffer.

### CLI changes: `src/wifiscope/cli.py`

Two new subcommands:

```
wifiscope monitor [--out path.jsonl] [--notify]
                  [--gateway IP] [--wan IP]
wifiscope calibrate [--duration 300]
```

`monitor`:
- No TUI. Runs `WiFiPoller` + `BLEPoller` + `LatencyPoller` +
  `EnvironmentMonitor` in parallel.
- Streams JSONL events to stdout; `--out` redirects to a file
  (append mode).
- `--notify` posts a macOS notification-centre alert via
  `osascript -e 'display notification ...'` for events with
  `confidence == "high"`. Subprocess call, no PyObjC.
- `--gateway IP` overrides the auto-detected gateway (rare; only
  useful when default-route detection fails).
- `--wan IP` overrides the auto-detected WAN anchor (replaces
  `WIFISCOPE_LATENCY_WAN_TARGET` for one-off invocations).
- DNS detection helper: a small standalone function in
  `latency.py` parses `scutil --dns` and returns the first
  non-gateway nameserver from `resolver #1`, or `None` when no
  such DNS is available. Tested in isolation against captured
  `scutil --dns` outputs (home / corporate / VPN / Tailscale
  shapes).

`calibrate`:
- Records 5 min (configurable) of RSSI baseline across all
  visible BSSIDs.
- Writes `./wifiscope-baseline.json`. Format: `{bssid: {rssi_mean,
  rssi_stddev, sample_count}}`.
- The Environment monitor reads this file at startup; if absent,
  uses the adaptive baseline.

## Known limitations to accept (do not solve)

- **Adaptive baseline drifts.** A user who leaves the office at
  6 PM and returns at 8 AM will have their baseline σ adapt
  downward overnight, then briefly fire false-positive events the
  next morning. We accept this; calibrate corrects it for users
  who care.
- **Multi-AP fusion misclassifies tiny offices.** A 1-AP home
  sees only the spatial-channel mode; redundancy fusion is unused.
  Confidence on those events stays "medium" instead of "high".
- **`/sbin/ping` precision is millisecond-only.** RTT < 1 ms (a
  modern wired LAN) reads as `0` or `1` because the binary does
  not surface microseconds. Not worth raw-socketing for sub-ms.
- **Loss-burst detection lags by up to 5 s.** The "3 of 5"
  rule means a single dropped packet is invisible until the
  next four samples decide.
- **Environment events are correlation, not causation.** A neighbour's
  AP rebooting can fire a stir event you did not cause; the OS
  refreshing its background scan cadence can fire a stir event
  no one caused. Do not promise more than "something changed".
- **WAN anchor probes the configured DNS, not Zoom / your real
  workload.** A healthy DNS does not guarantee Zoom works; a
  failed DNS does strongly suggest something is broken upstream
  of the gateway. We treat that as a fair trade for not having to
  ship a curated list of probe targets.
- **DNS auto-detection ignores DoH / DoT.** `scutil --dns` only
  surfaces the system-resolver IPs. Per-app encrypted-DNS
  configurations (Firefox DoH, Tailscale MagicDNS, etc.) are
  invisible — we ping whatever the OS resolver believes its
  upstream is. That is OK; the goal is link-state insight, not
  actual-application path verification.
